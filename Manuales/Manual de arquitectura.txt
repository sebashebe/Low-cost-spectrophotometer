\chapter{Visión General de la Arquitectura}\label{chap:intro}
% -----------------------------------------------------------------------------

\section{Propósito del Documento}
Este manual detalla la arquitectura interna del software de control desarrollado para el espectrofotómetro AS726X. Su objetivo es servir como una guía técnica exhaustiva para desarrolladores, con el fin de facilitar el mantenimiento, la depuración y la extensión de funcionalidades sobre la base de código existente. El documento asume que el lector posee conocimientos básicos de programación orientada a objetos y de Python.

\section{Filosofía de Diseño y Patrones}
La arquitectura del software se fundamenta en un conjunto de principios de diseño orientados a garantizar la robustez, mantenibilidad y escalabilidad del sistema.

La estructura principal se inspira en el conocido patrón \textbf{Modelo-Vista-Controlador (MVC)}, que promueve una clara separación de responsabilidades. El \textbf{Modelo}, representado por las \texttt{dataclasses} en \texttt{app\_state.py}, funciona como un contenedor pasivo para los datos y el estado de la aplicación. La \textbf{Vista}, compuesta por los componentes de la interfaz gráfica de PyQt5, se encarga exclusivamente de la presentación de la información y de la captura de las interacciones del usuario. En el centro, el \textbf{Controlador}, implementado en la clase principal \texttt{SpectrometerApp}, actúa como el intermediario que orquesta la comunicación entre la Vista y el Modelo, y ejecuta la lógica de la aplicación.

Este enfoque se complementa con un \textbf{diseño altamente modular}, donde cada funcionalidad importante está encapsulada en su propio módulo. La comunicación entre estos módulos se gestiona a través de un paradigma de \textbf{programación orientada a eventos}, utilizando el mecanismo de señales y slots de Qt. Esto crea un sistema desacoplado, donde los componentes emiten "señales" para notificar eventos, y otros componentes se "suscriben" a ellas a través de "slots", sin necesidad de dependencias directas.

Finalmente, para garantizar que la interfaz de usuario nunca se bloquee, las operaciones potencialmente lentas como la comunicación con el hardware se manejan de forma \textbf{asíncrona}. La lectura del puerto serie se delega a un hilo de ejecución secundario, que trabaja en segundo plano y notifica al hilo principal mediante señales cuando llegan nuevos datos, asegurando una experiencia de usuario siempre fluida.

\section{Jerarquía e Interacción entre Clases}

El componente central, que actúa como el orquestador de la aplicación, es la clase \textbf{\texttt{SpectrometerApp}}. Esta clase, que define la ventana principal, contiene las instancias de los demás componentes importantes: los paneles de la interfaz, los controladores de lógica como \texttt{FileManager} y \texttt{CalculationController}, y los objetos de estado del Modelo. Es el único componente con una visión global del sistema y la responsabilidad de coordinar todas las operaciones.

La interfaz de usuario (la Vista) está organizada bajo una jerarquía de herencia. Se definió una clase base, \textbf{\texttt{BasePanel}}, que actúa como una plantilla abstracta para todas las pestañas principales. Esta clase base establece una estructura visual común y una interfaz de comunicación estándar. Los paneles específicos como \texttt{MainSpectrumPanel} o \texttt{CalibrationCurveTab} heredan de \texttt{BasePanel} para implementar su funcionalidad particular. De esta forma se busca mantener la consistencia visual y funcional en toda la aplicación y la reutilización de código.

La interacción entre estos componentes se gestiona a través del \textbf{mecanismo de señales y slots de Qt}. En lugar de llamadas directas a métodos, los componentes se comunican de forma indirecta. Por ejemplo, cuando un usuario presiona un botón en un panel (Vista), este emite una señal (ej. \texttt{requestMeasurement}). La clase \texttt{SpectrometerApp} (Controlador) conecta previamente esa señal a uno de sus propios métodos (un slot), que ejecuta la lógica de medición. Este mismo mecanismo permite la comunicación segura entre hilos: el \texttt{SerialWorker}, desde su hilo secundario, emite la señal \texttt{data\_received}, que es gestionada por un slot en \texttt{SpectrometerApp} en el hilo principal de la interfaz.

\section{Decisiones Clave y Alternativas Consideradas}
La selección de tecnologías para el proyecto se basó en una evaluación de alternativas y sus respectivos compromisos (\textit{trade-offs}).

Se eligió \textbf{PyQt5} como framework para la interfaz de usuario sobre otras alternativas por su robusto conjunto de widgets y su excelente integración con \texttt{Matplotlib}, un requisito fundamental para la visualización de gráficos científicos.

Para el manejo de la concurrencia, se optó por el sistema de hilos nativo de Qt, \textbf{QThread}. Esta decisión se tomó por la integración nativa y sin fisuras con el bucle de eventos de PyQt5, ya que el uso de señales y slots entre hilos simplifica la arquitectura y reduce errores.

Finalmente, la estructura de la interfaz se diseñó de forma \textbf{modular por paneles}, donde cada pestaña es una clase independiente. Esto, si bien requiere una clase base y un sistema de señales, se alinea con el principio de Responsabilidad Única y facilita enormemente el mantenimiento y la escalabilidad del software a largo plazo.

\chapter{Guía de Despliegue y Dependencias}\label{chap:despliegue}
% -----------------------------------------------------------------------------
Esta sección detalla los requisitos de software y los pasos necesarios para configurar el entorno de desarrollo y ejecutar la aplicación.

\section{Dependencias de Software}
La aplicación está construida sobre Python y requiere varias librerías externas para funcionar.
\begin{itemize}[leftmargin=*]
    \item \textbf{Python:} Se recomienda la versión 3.8 o superior.
    \item \textbf{PyQt5:} Es el framework principal para la construcción de la interfaz gráfica de usuario.
    \item \textbf{Matplotlib:} Utilizada para la creación de todos los gráficos y visualizaciones de datos.
    \item \textbf{NumPy:} Es la librería fundamental para la computación numérica y el manejo de datos espectrales.
    \item \textbf{SciPy:} Utilizada específicamente para su módulo de estadísticas para calcular las curvas de calibración.
    \item \textbf{pyserial:} Proporciona la funcionalidad para la comunicación con el puerto serie del microcontrolador.
    \item \textbf{Arduino IDE:} Necesario para compilar y cargar el firmware (\texttt{Espectrofotometro.ino}) en la placa.
\end{itemize}

\section{Instalación del Entorno}
La forma recomendada de gestionar las dependencias es a través de un entorno virtual y el gestor de paquetes \texttt{pip}.
\begin{enumerate}
    \item \textbf{Crear un entorno virtual (Recomendado):}
    \begin{verbatim}
python -m venv venv
source venv/bin/activate  # En Windows: venv\Scripts\activate
    \end{verbatim}

    \item \textbf{Crear un archivo \texttt{requirements.txt}:} Cree un archivo de texto con este nombre en la raíz del proyecto y añada el siguiente contenido o use el archivo adjunto:
    \begin{verbatim}
PyQt5==5.15.10
matplotlib==3.8.4
numpy==1.26.4
scipy==1.13.1
pyserial==3.5
    \end{verbatim}
    
    \item \textbf{Instalar las dependencias:} Con el entorno virtual activado, ejecute el siguiente comando en la terminal:
    \begin{verbatim}
pip install -r requirements.txt
    \end{verbatim}
    
    \item \textbf{Ejecutar la aplicación:} Una vez instaladas las dependencias, la aplicación se puede iniciar ejecutando el script:
    \begin{verbatim}
python main.py
    \end{verbatim}
\end{enumerate}

\section{Controladores de Lógica> filemanager.py , calculationcontroller.py}

\begin{arquibox}[Clase \texttt{FileManager}]
Para mantener el código principal limpio y enfocado en la lógica de la aplicación, todas las operaciones de archivos se encapsularon en esta clase dedicada. Su única responsabilidad es manejar la lectura y escritura de datos en el disco.
\setlist{nolistsep}
\begin{description}[leftmargin=!,style=nextline]
    \item[\metodolabel{\_\_init\_\_(app)}] Guarda una referencia a la aplicación principal para poder acceder a los datos que necesita guardar.
    \item[\metodolabel{export\_log()}] Abre un diálogo de "Guardar como..." y escribe el contenido del log de eventos a un archivo de texto.
    \item[\metodolabel{export\_graph\_active\_tab()}] Guarda una imagen de alta resolución del gráfico de la pestaña visible.
    \item[\metodolabel{save\_session\_data()}] Guarda todos los datos de la sesión (referencia y mediciones) en un archivo CSV.
    \item[\metodolabel{save\_selected\_superimposed\_data()}] Guarda en un CSV únicamente los espectros que el usuario ha seleccionado en la lista de superposición.
    \item[\metodolabel{get\_save\_filename\_for\_sequential()}] Pide al usuario la ubicación para guardar los resultados de una medición secuencial antes de que comience.
    \item[\metodolabel{save\_sequential\_data(m\_state)}] Escribe los resultados de una medición secuencial a un archivo CSV.
    \item[\metodolabel{\_get\_active\_figure\_and\_prefix()}] Método interno que determina qué pestaña está activa para saber qué gráfico exportar.
    \item[\metodolabel{\_get\_save\_filename(title, ...)}] Abre un diálogo de "Guardar como..." estándar, sugiriendo un nombre de archivo con timestamp.
    \item[\metodolabel{\_write\_csv(filename, data\_lines, ...)}] Función interna que maneja la escritura de datos a un archivo CSV de manera formateada.
\end{description}
\end{arquibox}

\begin{arquibox}[Clase \texttt{CalculationController}]
Esta clase encapsula toda la lógica matemática para el cálculo de concentración. Fue aislada para que el código de la clase principal fuera más legible y para tener toda la matemática en un solo lugar.
\setlist{nolistsep}
\begin{description}[leftmargin=!,style=nextline]
    \item[\metodolabel{\_\_init\_\_(app)}] Guarda una referencia a la aplicación principal para acceder a los datos del espectro y a los parámetros de calibración activos.
    \item[\metodolabel{update\_display\_only()}] Calcula la concentración con los datos actuales y actualiza las etiquetas de la UI en tiempo real, sin registrar el dato.
    \item[\metodolabel{calculate\_and\_register()}] Calcula la concentración y guarda el espectro actual como una nueva medición en la lista de la sesión.
    \item[\metodolabel{\_calculate\_concentration\_core()}] Es la función central de la clase. Implementa la lógica para decidir qué método de cálculo usar (Modelo de Transferencia, Curva de Sesión o Ley de Beer-Lambert) y aplica la matemática correspondiente.
    \item[\metodolabel{\_get\_current\_absorbance\_spectrum()}] Calcula el espectro de absorbancia actual ($A = -\log_{10}(I/I_0)$) a partir de la intensidad de la muestra y de la referencia.
\end{description}
\end{arquibox}

\section{Comunicación Asíncrona: \texttt{workers.py}}
Este módulo es crucial para que la aplicación sea fluida. Define el "trabajador" que se ejecuta en un hilo separado para manejar la comunicación serial.

\begin{arquibox}[Clase \texttt{SerialWorker(QObject)}]
Su única misión es leer continuamente el puerto serie sin bloquear la interfaz. Cuando recibe un dato, emite una señal para notificar al hilo principal, manteniendo la UI siempre responsiva.
\setlist{nolistsep}
\begin{description}[leftmargin=!,style=nextline]
    \item[\metodolabel{\_\_init\_\_(serial\_instance)}] Recibe la instancia del puerto serial que fue abierta en la clase principal.
    \item[\metodolabel{run()}] Es el bucle principal que se ejecuta en el hilo secundario. Mientras esté activo, lee los datos del puerto y emite la señal \texttt{data\_received} con cada nueva línea.
    \item[\metodolabel{stop()}] Se llama desde el hilo principal para detener al worker de forma segura, finalizando el bucle de lectura.
\end{description}
\end{arquibox}

\section{El Firmware: \texttt{Espectrofotometro.ino}}
Este es el código que se ejecuta en el microcontrolador Arduino. Su función es ser el puente de bajo nivel entre el software y el hardware físico (sensor y LED). Fue diseñado para ser simple y reactivo.
\begin{description}[leftmargin=!,style=nextline]
    \item[\metodolabel{setup()}] Se ejecuta una vez al iniciar el Arduino. Configura la comunicación serial, inicializa el sensor y prepara el pin del LED.
    \item[\metodolabel{loop()}] Es el corazón del firmware, se ejecuta continuamente. Revisa si ha llegado un comando del software y, si está en modo de medición, toma una lectura y la envía.
    \item[\metodolabel{processCommand(cmd)}] Parsea los comandos que llegan por serial ("calibrar", "empezar", "detener") y cambia el estado del Arduino.
    \item[\metodolabel{printMeasurements(prefix)}] Toma la lectura de los 6 canales del sensor, la formatea como una cadena de texto y la envía por el puerto serie al software.
    \item[\metodolabel{setWhiteLEDOn() / setWhiteLEDOff()}] Funciones simples para encender y apagar el LED.
\end{description}

\chapter{El Modelo: Gestión del Estado y los Datos}\label{chap:modelo}

Esta capa define la estructura de todos los datos de la aplicación. Para esto, se utilizan \texttt{dataclasses} de Python, que permiten crear contenedores de estado claros y organizados. Esta es la "única fuente de verdad" de la aplicación, lo que evita que el estado se disperse y facilita la depuración.

\section{Estructuras de Estado Centralizadas: \texttt{app\_state.py}}
Este archivo no contiene lógica, solo la definición de la estructura de los datos.

\begin{notebox}[Dataclasses de Estado]
Las siguientes clases son contenedores de datos. Su única función es almacenar y organizar el estado de la aplicación de una manera limpia y predecible.
\setlist{nolistsep}
\begin{description}[leftmargin=!,style=nextline]
    \item[\metodolabel{PlotOptionsState}] Almacena las opciones de visualización de los gráficos: qué espectros superponer, si la escala es logarítmica, si se muestran marcadores y el tipo de gráfico actual.
    \item[\metodolabel{MeasurementDataState}] Contiene los datos de la sesión actual: las lecturas de referencia (I\textsubscript{0}) y de muestra (I), y si la sustracción de blanco está activa.

\end{notebox}

Esta clase base abstracta fue creada para que todos los paneles principales (pestañas) heredaran de ella. De esta forma, se asegura que todos tengan una estructura y una forma de comunicarse consistentes, evitando duplicar código.

\chapter{Módulos de Soporte y Constantes}\label{chap:soporte}
% -----------------------------------------------------------------------------
Estos módulos no contienen lógica compleja, pero son fundamentales para la consistencia y el mantenimiento del código.

\section{Constantes Globales: \texttt{constants.py}}
Este archivo centraliza todos los valores fijos. Esta práctica es crucial para el mantenimiento: si se necesita cambiar un color o un texto, solo se debe modificar en este archivo en lugar de buscarlo en todo el proyecto.
\begin{itemize}[leftmargin=*]
    \item \textbf{Parámetros del Sensor:} Las longitudes de onda del sensor y sus etiquetas para la UI.
    \item \textbf{Paletas de Colores:} Dos diccionarios que definen los colores para los gráficos en los temas oscuro y claro.
    \item \textbf{Hojas de Estilo (QSS):} Dos grandes strings que contienen el código CSS que le da estilo a toda la interfaz gráfica.
    \item \textbf{Textos de Ayuda:} Un diccionario que contiene todos los textos para los diálogos de ayuda contextual.
\end{itemize}

\section{Tipos y Enumeraciones: \texttt{measurement\_type.py}}
Este archivo usa enumeraciones de Python para definir constantes con nombre. Esto mejora la legibilidad y previene errores de tipeo.
\begin{itemize}[leftmargin=*]
    \item \textbf{Clase `MeasurementType`:} Define los tipos de medición: \texttt{CONTINUOUS}, \texttt{SINGLE\_AVERAGED}, \texttt{SEQUENTIAL}.
    \item \textbf{Enum `PlotType`:} Define los tipos de gráficos: \texttt{INTENSITY}, \texttt{ABSORBANCE}, \texttt{TRANSMITTANCE}.
    \item \textbf{Enum `AppState`:} Define los estados principales de la aplicación: \texttt{IDLE}, \texttt{MEASURING\_MAIN}, \texttt{MEASURING\_DIRECT}, \texttt{CLOSING}.
\end{itemize}
